Travelpedia — Technical overview

For: New web developers learning how this project works.

1. What the app does

Travelpedia is a tour marketplace: operators list trips, visitors browse active listings and send inquiries, and admins approve operators and tours before they go live.

2. Technology stack

There is no separate custom API server in this repo: Next.js calls Supabase’s APIs using the official clients.

3. Where is the database?

The database is not a file in your project. It lives in the cloud on your Supabase project (PostgreSQL).

  • You manage tables and SQL in the Supabase Dashboard.
  • The app connects using NEXT_PUBLIC_SUPABASE_URL and NEXT_PUBLIC_SUPABASE_ANON_KEY in .env.local (see .env.example for variable names).
  • The anon key is public in the browser by design; Row Level Security (RLS) in Postgres defines who can read or write what.

Main tables

Defined in supabase/migrations/001_initial_schema.sql:

  • profiles — one row per user; links to auth.users; stores role (operator/admin), approval_status, company info.
  • tours — listings; status is pending / active / closed.
  • inquiries — visitor messages about a tour.

4. Frontend vs backend in this project

  • Frontend (browser): React client components ("use client") for forms and interactivity; they use src/lib/supabase/client.ts.
  • Backend (server): Server Components and layouts use src/lib/supabase/server.ts with cookies to know who is logged in. Server Actions in src/app/actions/ perform admin/operator updates after checking permissions.
  • Middleware (src/middleware.ts) refreshes the Supabase session on requests. Admin/operator routes are enforced mainly in layouts (redirect if not allowed).

Public tour listings use fetchActiveTours() in src/lib/tours-query.ts with an anonymous server client — RLS only allows reading status = 'active' tours.

5. How to access the admin account

Registration always creates an operator with approval_status = 'pending'. You cannot become admin through the sign-up form.

A database trigger creates each profile as operator; admins are promoted manually.

  1. Create a user in Supabase Authentication → Users (or register via the app and confirm email if required).
  2. In SQL Editor, run (replace the UUID with that user’s id from Authentication → Users):
update public.profiles
set role = 'admin', approval_status = 'approved'
where id = '<auth-user-uuid>';

Then sign in at /auth/login. The login form sends users with role === 'admin' to /admin.

src/app/admin/layout.tsx requires an approved admin profile; others are redirected to login.

6. How email verification works

Handled by Supabase Auth, not custom email code in this repo.

src/app/auth/register/register-form.tsx calls supabase.auth.signUp(). If your Supabase project has Confirm email enabled (Authentication → Providers → Email), Supabase sends a confirmation link. Users must confirm before sign-in, depending on your settings.

For local testing, you can confirm users in the dashboard or adjust email settings; re-enable confirmation for production.

7. Operator and tour approval

  • Operators stay pending until an admin approves them in /admin/operators (see src/app/actions/admin.ts). RLS blocks tour creation until the operator is approved.
  • New tours are pending until an admin sets them active in /admin/tours; only active tours appear on the public site.

8. Running locally

  1. npm install
  2. Copy .env.example to .env.local and add your Supabase URL and anon key.
  3. Run the SQL migration in Supabase if needed.
  4. npm run dev — open the local URL (often http://localhost:3000).

9. Security notes

  • Do not disable RLS without understanding the impact.
  • Keep real secrets in .env.local (not committed).
  • Service role keys (if ever used) must never ship to the browser.
